# Project Library of Babel (v0.2)

## 1. 概要
`Project Library of Babel` は、情報を保存するのではなく、決定論的なハッシュ関数（BLAKE3）によって生成される無限の疑似乱数空間（辞書空間）の中の「座標」として再定義する、実験的な暗号化・エンコードシステムです。

物理的なデータの断片を、ハッシュ関数の計算過程における出現位置（スキップ回数 $M$ とハッシュ内開始インデックス $I$）に置換することで、情報の局所性を完全に排除し、空間上の点へと変換します。

v0.2 では、並列処理を可能にするためチャンクごとの独立シード生成への移行、超コンパクトなバイナリアーカイブ形式（Golomb-Rice符号 + 適応型 Order-1 レンジコーダー）、およびサイズ見積もりを行う `analyze` コマンドが実装されました。

---

## 2. 構成要素
本システムは以下の4つの動的要素によって定義されます。

| 要素 | 定義 | 説明 |
| :--- | :--- | :--- |
| **S (Salt)** | 公開値 | システムが生成する一意の16バイト乱数。辞書空間に個体差（ユニーク性）をもたらす。 |
| **P (Password)** | 秘密鍵 | ユーザーが管理する暗号鍵。辞書空間へのアクセス権限（決定論的なシードの源泉）となる。 |
| **H(x)** | ハッシュ関数 | **BLAKE3** を使用。高速かつ極めて決定論的な256bit出力を利用。 |
| **D (Dictionary)** | 辞書空間 | 初期シードからハッシュ関数 $H(x)$ を再帰的に適用することで生成される、事実上無限のバイト列。 |

---

## 3. アルゴリズム (The Process)

v0.2 からは、各チャンクの処理を前後の依存性なく完全並列（マルチスレッド）で行えるように改良されました。

### Step 1: 初期シード生成 (並列化対応)
元データを指定された `ChunkSize` $C$ （$1 \sim 4$ バイト）のチャンク $c_0, c_1, ..., c_n$ に分割します。
各チャンク $j$ のハッシュ生成の起点となる独立した初期シード値 $G_j$ を決定論的に生成します。
$$G_j = H(P + S + \text{to\_le\_bytes}(j))$$

### Step 2: スキャンとマッチング
各チャンク $j$ について、シード $G_j$ から始まるハッシュの走査を開始します。

1. **照合 (Matching)**:
   現在のハッシュ値 $H_i$ （32バイト）の内部に、チャンクデータ $c_j$ と完全に一致するバイトパターン（連続する $C$ バイト）があるかスライド窓で走査します。一致可能な位置（インデックス $I$）の範囲は $0 \le I \le 32 - C$ です。

2. **失敗時 (No Match)**:
   - 空振り回数（`misses`）をインクリメントします。
   - ハッシュを更新します: $H_{i+1} = H(H_i)$
   - 一致するパターンが見つかるまで、この探索ループを繰り返します。

3. **成功時 (Match)**:
   - 一致したハッシュ内の開始位置インデックス $I$（$0 \sim 31$）と、それまでの空振り回数 $M$ (`misses`) を座標として記録します： `MatchCoord { misses: M, index: I }`

## 4. アーカイブ構造 (Output Format)

本システムは、2種類のアーカイブ出力に対応しています。

### 4.1 テキスト形式 (`.babel.txt`)
人間が読めるレガシー形式です。
```ini
[Header]
Version: ABGRUND INDEX 0.1
Hash: BLAKE3-256
Salt: <16進数 32文字>
ChunkSize: <数値>

[Body]
<座標リスト: x14_0.0, 7, 22, x8_0.0, 3, ...>

[Footer]
Checksum: <元データのBLAKE3ハッシュ>
```

### 4.2 バイナリ形式 (`.babel.bin`)
v0.2 で追加された、エントロピー極限までサイズを圧縮する高密度バイナリフォーマットです。

```text
[ 4 bytes ]  Magic: b"BAB\x01"
[ 1 byte  ]  ChunkSize (1..=4)
[ 16 bytes]  Salt
[ 32 bytes]  元データのBLAKE3チェックサム（生のバイナリ）
[ 4 bytes ]  ANS-coded payload のバイト長 (u32 LE)
[ N bytes ]  ANS-coded payload (以下で詳述)
```

#### 【ANS-coded payload の圧縮プロセス】
1. **Rice符号化 (Golomb-Rice coding)**:
   幾何分布に従う空振り回数（`misses`）を Rice符号を用いて符号化（可変長ビットの商 `q` (Unary) + 剰余 `r`）し、インデックス $I$ は固定 5-bit integer としてビットパックします。
2. **Order-1 レンジコーダー (Adaptive Range Coder)**:
   ビットパックされたバイトストリームに対し、適応型 **Order-1 context model** を用いた **range-coder** （算術符号化の変種）を施すことで、ビットパック後に残存する構造的依存性のエントロピーを除去し、最適な物理圧縮を実現します。

## 5. 復号ロジック (Decryption)

復号のプロセスは暗号化の決定論的ステップをなぞることで、オンメモリで高速に行われます。

1. **初期化**: パスワード $P$ とヘッダーの $S$ から、暗号化時と全く同一の初期シード $G_j$ をチャンクごとに計算します。
2. **ハッシュの再生**: 各座標 `MatchCoord { misses: M, index: I }` に対し、ハッシュ計算を $M$ 回進めて $H_M$ に到達させます。
3. **データ抽出**: ハッシュ $H_M$ のインデックス $I$ から `ChunkSize` 分のバイトデータを抽出して復元します。
4. **検証**: 結合された復号データの BLAKE3 ハッシュを計算し、`Checksum` と一致するかを検証します。

## 6. クイックスタートガイド (Quick Start Guide)

### 1. インストール / ビルド
Rust（`cargo`）がインストールされている環境でビルドします。
```bash
cargo build --release
```

### 2. 暗号化 (Encrypt)
ファイル、パスワードを指定して暗号化します。
```bash
# デフォルトでバイナリ圧縮形式 (.babel.bin) で暗号化
./target/release/library_of_babel_crypt encrypt secret.txt mypassword123 1 --binary

# テキスト形式 (.babel.txt) での出力も可能
./target/release/library_of_babel_crypt encrypt secret.txt mypassword123 1 --text
```
* 実行中は、超高速にリフレッシュされる TUI（ターミナルUI）により、ハッシュレートや走査座標、進捗状況がリアルタイムで可視化されます。

### 3. 復号 (Decrypt)
```bash
./target/release/library_of_babel_crypt decrypt secret.txt.babel.bin mypassword123
```
* アーカイブのヘッダー（Magic）を判別し、正しいデコーダが自動選択されて元のデータ `secret.txt.dec` が復元されます。

### 4. 期待サイズ分析 (Analyze)
入力ファイルに対して、チャンクサイズごとのハッシュ計算量や出力ファイルサイズ（テキスト/バイナリRAW/バイナリANS）の推定値を瞬時にシミュレーション表示します。
```bash
./target/release/library_of_babel_crypt analyze secret.txt
```

## 7. 【重要】現実的な限界と考察（圧縮について）

本プロジェクトは、**「情報を保存するのではなく、無限のハッシュ空間の座標に置換する」** という極めて数学的でロマンのあるアプローチを採用していますが、**「実用的な『データ圧縮ツール』として機能させることは現実的に極めて厳しい（実質的に不可能）」** です。これには明確な二つの数学的・計算量的な壁が存在します。

### 7.1 「データ肥大化の壁」（チャンクサイズが小さい場合）
`ChunkSize` を最小の `1 byte` に設定した場合、32バイトのハッシュ内に1バイトが一致する確率は非常に高く、走査は一瞬（平均8.5回のハッシュ計算）で終わります。
しかし、**わずか 1バイト（8ビット）の元データを表現するために、座標情報である `misses`（Golomb-Rice可変長）と `index`（5ビット）を記録しなければなりません。**
どれほど高度なエントロピー圧縮（Order-1 レンジコーダー）を施しても、座標のメタデータ量は平均して 1バイト を超えてしまい、暗号化した結果は元のファイルサイズより **必ず肥大化（通常 1.2〜1.5倍）します。**

### 7.2 「ハッシュエクスプロージョンの壁」（チャンクサイズが大きい場合）
データサイズを削減し、「奇跡の超圧縮」を達成するためには、`ChunkSize` を大きくして（例: 4bytes以上）1つの座標で表現する元データを増やす必要があります。
しかし、チャンクサイズを大きくすると、ハッシュ内でパターンが一致する確率は**指数関数的（幾何分布）に激減します。**

1チャンクを発見するために必要なハッシュ計算回数の期待値は以下の通りです：

| チャンクサイズ | 平均ハッシュ計算回数 | 計算時間（目安） | 圧縮率の期待 |
| :--- | :--- | :--- | :--- |
| **1 byte** | $\approx 8.5$ 回 | ほぼ一瞬 | 肥大化 (120%〜150%) |
| **2 bytes** | $\approx 2,114$ 回 | 一瞬 | やや肥大化〜トントン |
| **3 bytes** | $\approx 559,240$ 回 | 数ミリ秒 | ほぼ互角 |
| **4 bytes** | $\approx 1.48 \times 10^8$ 回 | 数秒 | やや圧縮できる可能性 |
| **8 bytes** | $\approx 7.37 \times 10^{17}$ 回 | **宇宙年齢レベル** | 劇的な超圧縮 |

チャンクサイズが 4バイトの場合、1チャンクをマッチさせるだけで平均約 1億5000万回 のハッシュ計算が必要です。
さらに 8バイト のチャンクをハッシュ探索で見つけようとすれば、計算回数は天文学的な $7.3 \times 10^{17}$ 回に達し、地球上の全スーパーコンピュータを動員しても一生終わらない処理時間になります。

### 結論
本システムは、「情報そのものを消去し、ハッシュ宇宙の住所（座標）として表現する」という哲学的・暗号論的に美しいコンセプトの実験作です。しかし、**「計算量を削減しようとするとデータサイズが増大し、データサイズを削減しようとすると計算量が無限に爆発する」** というトレードオフから逃れられず、実用的な圧縮ツールとしては破綻しています。

この極限状態を楽しむためのシステムこそが、この **`Library of Babel Crypt`** です。

なお、この「計算量とデータサイズの等価壁」を突破し、現実的な処理時間での実用的なデータ圧縮を実現するための実験的アプローチ（多層ハッシュや動的ソルト誘導の最適化など）については、メインのコードベースから分離し、**別ブランチにて引き続き粘り強く研究・開発を行っていきます。**
